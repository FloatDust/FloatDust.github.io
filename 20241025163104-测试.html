测试

测试三要素

  1. 设置输入
  2. 运行测试代码
  3. 判断结果(assert)

断言

assert_eq!

assert_eq!(a, b) 测试是否相等

  • 其中 a 与 b 都必须实现 PartialEq 和 Debug 特征
  • 可以为自定义的结构体、枚举添加 #[derive(PartialEq, Debug)] 注解

assert_ne!

测试不等

assert!

assert!(true)

debug_assert!

只能在 debug 模式( cargo run )下输出,在 cargo run --release 时没有输出

  • debug_assert_eq! debug_assert_ne! debug_assert!

单元测试

#[cfg(test)]
mod tests {
    #[test]
    fn it_works() {
        assert_eq!(2 + 2, 4);
    }
}

特点

  1. 注解 #[cfg(test)]
  2. 模块名 tests
  3. 测试函数属性 #[test]
  4. 添加了 test 注解的模块和函数,就不能编译到 crate 中了,也不能引用

自定义失败信息

在 assert 后面添加额外参数

assert!(
    result.contains(target),
    "你的问候中并没有包含目标姓名 {} ，你的问候是 `{}`",
    target,
    result
)

panic

测试出错边界 #[should_panic]

  • expected

    expect 中的信息必须匹配出错信息

    panic!(
    "Guess value must be less than or equal to 100, got {}.",
    value
);

#[should_panic(expected = "Guess value must be less than or equal to 100, got")]

Result<T, E>

使用 Result 替换 assert

#[ignore]

}

额外包

pretty_assertions:提供彩色字体结果对比

集成测试

  • 在根目录下建立 tests 目录,不需要加 #[cfg(test)] 注解了
  • tests 目录下的子目录的文件不会被当作测试文件
    • 在测试文件中建立共同函数文件,引用时也要加 mod common

基准测试

性能测试分为

  • 压力测试
    用户大量访问
  • 性能测试
    函数运行速度

基准测试就是性能测试

官方

#[bench]
fn bench_add_two(b: &mut Bencher) {
    b.iter(|| add_two(2));
}
  • rust 需要 nightly 版本
  • 添加 #[bench] 属性
  • 使用 cargo bench 命令

criterion = "0.3"
